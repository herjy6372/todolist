# 🚀 나의 할 일 (MyTodo)
> **개발 기간:** 2026.04 ~ 진행중 (개인 프로젝트, 1인 개발)
> **배포 링크:** Google Play Store 비공개 테스트 중
> **앱 버전:** 1.4.2 (32)

---

## 💡 1. 프로젝트 소개

**"나의 할 일"** 은 단순 할 일 관리를 넘어, 캘린더·기분 기록·가계부·건강 관리·세계시각·타이머까지 아우르는 **개인용 올인원 라이프 관리 Flutter 앱**입니다.

로그인 없이도 로컬 저장소만으로 모든 기능을 오프라인 사용할 수 있게 설계했고, 로그인 시에는 Firebase 기반 클라우드 동기화로 여러 기기 간 데이터를 이어서 사용할 수 있습니다. 또한 무료(일반)/학생/프로/베타/관리자 5단계 등급 체계를 직접 설계해, 기능 단위로 접근 권한을 세밀하게 통제하는 구조를 구현했습니다.

혼자 기획부터 설계, 개발, 배포 준비(Play Store 심사 대응, iOS 빌드 환경 트러블슈팅)까지 전 과정을 진행하고 있는 프로젝트입니다.

## 🛠 2. 사용 기술 및 개발 환경 (Tech Stack)

**Language / Framework**
- Dart, Flutter (SDK ^3.11.5)

**Backend / Auth**
- Firebase Core / Firebase Auth / Cloud Firestore
- Google Sign-In (소셜 로그인)

**로컬 저장 & 상태 관리**
- `shared_preferences` 기반 로컬 캐시 (오프라인 우선 저장)
- `ChangeNotifier` 기반 Repository 패턴 (별도 상태관리 라이브러리 없이 자체 구현)

**알림 / 위치 / 외부 API**
- `flutter_local_notifications` + `timezone` (일일 로컬 푸시 알림)
- `geolocator` + `http` (Open-Meteo 기반 실시간 날씨 조회)
- 공공데이터포털 특일정보 API (한국 공휴일), `holiday_jp` (일본 공휴일 오프라인 DB)

**UI/UX**
- `table_calendar` (캘린더), `fl_chart` (기분/가계부 통계 그래프)
- `flutter_localizations` 기반 5개 언어 지원 (한국어/영어/일본어/필리핀어/베트남어)
- 다크모드 + 4종 커스텀 테마(고양이/강아지/기린/커피, 테마별 패턴 배경)

**빌드 / 배포**
- `flutter_launcher_icons`, `flutter_native_splash` (아이콘·스플래시 자동 생성)
- iOS: Xcode + CocoaPods, Android: Gradle (Play Console 공개 테스트 트랙 준비)

## 🌟 3. 핵심 기능 및 구현 내용

### 🔑 주요 기능
* **할 일 관리** — 카테고리(업무/공부/생활/건강/취미/기타)별 분류, 캘린더 뷰에서 한국·일본 공휴일 자동 표시
* **등급별 기능 접근 제어(RBAC)** — `일반 / 학생 / 프로 / 베타 / 관리자` 5단계 권한 모델을 enum으로 정의하고, 화면 단위가 아닌 **기능 단위**로 접근을 통제 (예: 클라우드 동기화는 베타·관리자만, 동물 테마는 프로 이상만)
* **오프라인 우선 + 선택적 클라우드 동기화** — 로그인하지 않아도 로컬로 전체 기능 사용 가능하며, 로그인 시 Firestore와 동기화. `SyncBackend` 추상 인터페이스로 `NoopSyncBackend`(로컬 전용)와 `FirebaseSyncBackend`(클라우드)를 전환 가능하게 설계
* **날씨 연동 일일 알림** — 위치 기반 실시간 날씨를 조회해 알림 문구에 반영하고, 날씨 로드가 끝나면 알림을 재예약하는 비동기 파이프라인 구성
* **건강/가계부/기분 트래킹** — 걸음 수, 수분·복약 체크, 체중/BMI, 가계부, 기분 기록을 각각 저장소로 분리하고 `fl_chart`로 주간 통계 시각화
* **보호자 동의 & 프로 등급 승급 신청** — 만 14세 미만 학생 계정은 베타/프로 등급 부여를 제한하고, 일반 사용자는 관리자에게 프로 등급 승급을 신청 → 관리자가 Firestore 대시보드성 화면에서 승인/거절하는 워크플로 구현
* **공지사항/업데이트 내역 관리** — 관리자만 작성 가능한 공지/업데이트 게시판을 Firestore 스트림으로 실시간 반영
* **다국어 + 다크모드 + 커스텀 테마** — 5개 언어와 라이트/다크, 4종 캐릭터 테마를 앱 전역에서 실시간 전환

## 🪵 4. 아키텍처 및 시스템 구조

* **Repository 패턴 + ChangeNotifier** — `TodoRepository`, `MoodRepository`, `LedgerRepository`, `ProfileRepository` 등 도메인별 저장소를 싱글턴으로 분리하고, `ChangeNotifier`로 UI에 변경을 알리는 경량 상태관리 구조 (외부 상태관리 라이브러리 미사용)
* **Strategy 패턴 기반 동기화 계층** — `abstract class SyncBackend` 인터페이스 아래 `NoopSyncBackend`(로컬)와 `FirebaseSyncBackend`(클라우드) 구현체를 두어, 로그인 여부·등급에 따라 런타임에 백엔드를 교체
* **`main()` 초기화 파이프라인** — Firebase 초기화 실패 시에도 앱이 죽지 않도록 `try/catch`로 감싸고 로컬 전용 모드로 폴백, 다국어 날짜 포맷 초기화 → 저장소 로드 → 알림 서비스 초기화 → 날씨는 UI 렌더링 이후 비동기 로드하여 앱 시작 지연을 최소화
* **등급(Role) 기반 권한 게이트** — UI 컴포넌트가 아니라 `UserRole` enum의 getter(`canSync`, `canUsePro`, `canUseBeta`, `canEditAnnouncements` 등)에 권한 로직을 응집시켜, 화면 코드는 단순히 getter만 참조하도록 설계

## 🔥 5. 트러블 슈팅 (Troubleshooting) — ★가장 중요★

### ❌ 문제 상황
최신 iOS 시뮬레이터(Xcode 27 베타, macOS 27.0 베타) 환경에서 `flutter run` 실행 시 아래와 같은 오류로 빌드가 계속 실패했습니다.

```
Target Integrity (Xcode): The iOS Simulator deployment target 'IPHONEOS_DEPLOYMENT_TARGET'
is set to 13.0, but the range of supported deployment target versions is 15.0 to 27.0.x.
...
Failed to build iOS app
Uncategorized (Xcode): Exited with status code 255
```

배포 타겟을 15.0으로 올린 뒤에도 `Failed to copy Flutter framework` 로 계속 실패했고, 로그를 추적한 결과 실제 원인은 다음과 같았습니다.

```
Binary .../Flutter.framework/Flutter does not contain architectures "arm64 x86_64".
lipo -info:
Architectures in the fat file: .../Flutter are: x86_64 arm64
```

`lipo -info` 로는 두 아키텍처가 분명히 존재하는데도 Flutter 빌드 스크립트는 계속 "아키텍처가 없다"고 실패하는 모순된 상황이었습니다.

### 🔍 원인 및 해결
1. **1차 원인 — 배포 타겟 하드코딩**: `Podfile`과 `Runner.xcodeproj`에 `IPHONEOS_DEPLOYMENT_TARGET = 13.0`이 하드코딩돼 있어 새 Xcode의 최소 지원 버전(15.0)에 걸림 → 두 파일 모두 15.0으로 수정.
2. **2차 원인 — 깨진 CocoaPods 심볼릭 링크**: `ios/.symlinks/plugins/cloud_firestore` 항목이 깨져 있어 `pod install` 자체가 `Invalid Podfile` 오류로 실패 → `.symlinks`, `Pods`, `Podfile.lock` 을 삭제하고 `flutter clean` → `flutter pub get` → `pod install` 순으로 재생성해 해결.
3. **근본 원인 — `lipo` 툴 자체의 동작 변경**: 직접 `lipo <binary> -verify_arch arm64 x86_64`를 실행해보니 `lipo: -verify_arch requires exactly one input file`라는 에러가 발생했습니다. 즉, 이번 Xcode 27 베타에 포함된 `lipo`가 `-verify_arch`에 아키텍처를 **한 번에 하나만** 받도록 동작이 바뀌었는데, 현재 Flutter(3.44.2)의 빌드 스크립트는 여전히 `lipo -verify_arch arm64 x86_64`처럼 여러 개를 한 번에 넘기고 있어 매번 실패한 것이었습니다.
   - `EXCLUDED_ARCHS`로 x86_64를 제외하는 일반적인 방법을 먼저 시도했지만, Run Script 단계에 전달되는 `$ARCHS` 환경변수 값 자체는 `EXCLUDED_ARCHS`로 걸러지지 않는다는 것을 `xcodebuild -showBuildSettings` 로 직접 확인.
   - 최종적으로 `Runner.xcodeproj`의 프로젝트/타겟 빌드 설정과 `Podfile`의 `post_install` 훅 양쪽에 **`ARCHS[sdk=iphonesimulator*] = arm64`** 를 명시적으로 주입해, 시뮬레이터 빌드 시 `$ARCHS`가 애초에 `arm64` 단일 값으로 해석되도록 강제 → `lipo -verify_arch arm64` 단일 인자 호출이 되면서 빌드 정상화.

이 과정에서 `EXCLUDED_ARCHS`와 `ARCHS`가 Xcode 빌드 시스템 내부적으로는 서로 다른 시점에 적용되며(전자는 실제 컴파일 대상 필터링, 후자는 Run Script에 노출되는 값 자체), 표면적으로 같은 문제를 해결하는 것처럼 보여도 실제로는 전혀 다른 레이어에서 동작한다는 점을 실측을 통해 확인하고 대응했습니다.

## 📈 6. 성과 및 회고
* 상태관리 라이브러리 없이 `ChangeNotifier` + Repository 패턴만으로 10개 이상의 도메인(할일/기분/가계부/세계시각/프로필/테마/언어/공지/알림/보호자동의)을 관리 가능한 구조로 분리해본 경험을 쌓았습니다.
* 오프라인 우선 설계(`NoopSyncBackend`)와 클라우드 동기화(`FirebaseSyncBackend`)를 인터페이스로 분리해 두어, 이후 백엔드를 교체하거나 테스트 환경을 구성할 때 기존 코드를 건드리지 않고 확장할 수 있는 구조를 실제로 검증했습니다.
* 등급 기반 권한 모델을 화면이 아닌 도메인 모델(enum getter)에 응집시킨 덕분에, 새로운 유료/베타 기능을 추가할 때마다 화면 코드를 뒤지지 않고 `UserRole` 한 곳만 수정하면 되는 구조의 이점을 체감했습니다.
* 이번 iOS 빌드 트러블슈팅을 통해, 표면적인 에러 메시지(`does not contain architectures`)만 보고 넘어가지 않고 `lipo`, `xcodebuild -showBuildSettings` 를 직접 실행해 Xcode 빌드 시스템의 실제 동작을 검증하는 습관의 중요성을 다시 확인했습니다. 특히 베타 OS/Xcode 조합에서는 공식 문서에 없는 툴체인 동작 변화가 생길 수 있다는 점을 실무적으로 경험했습니다.
* 앞으로는 Play Store 공개 테스트 출시, 개인정보 처리방침 정비, 등급별 결제(인앱결제) 연동을 다음 단계 과제로 진행할 예정입니다.
